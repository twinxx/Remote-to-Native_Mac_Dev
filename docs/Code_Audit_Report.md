# 🚨 코드 감사 및 취약점 분석 보고서 (Code Audit Report)

행님의 지적에 따라 현재 프로젝트의 주요 스크립트(`server.ts`, `receiverScript.ts`, `start.sh/bat`)들을 원점에서부터 철저하게 다시 분석했습니다.
기능 구현에만 급급하여 놓쳤던 수많은 예외 처리 부재와 보안 취약점, 그리고 구조적 결함들을 발견했습니다. 진심으로 반성하며 상세 내용을 보고드립니다.

## 1. `server.ts` 의 주요 결함 및 개선점

### 🔴 A. 보안 취약점: Path Traversal (디렉토리 경로 우회) 위험
*   **문제점:** `/api/files/:project` 엔드포인트나 파일 다운로드 로직에서 `:project` 파라미터를 검증 없이 그대로 `join()`에 사용하고 있습니다. 악의적인 사용자(또는 버그)가 `../../../` 와 같은 문자열을 보내면 프로젝트 폴더를 벗어나 시스템의 민감한 파일에 접근할 수 있습니다.
*   **대책:** 들어오는 파라미터나 경로 값에 대해 `path.normalize()` 및 상위 폴더 접근 문자열(`..`) 차단 등 강력한 입력값 검증(Sanitization) 로직이 도입되어야 합니다.

### 🔴 B. 파일 시스템 접근 시 충돌 및 예외 처리 부재
*   **문제점:** `readFileSync`, `writeFileSync`, `readdirSync` 등 동기식 파일 입출력을 남발하면서도 `try...catch`로 감싸지 않은 곳이 너무 많습니다 (예: API 라우트 내부). 파일이 쓰기 잠금 상태이거나 권한이 없을 경우 서버 전체가 크래시(Crash)되며 죽어버립니다.
*   **대책:** 
    1. 가능하면 `fs.promises` 를 활용한 비동기 I/O를 사용하여 Node.js 이벤트 루프 블로킹을 방지해야 합니다.
    2. 모든 I/O 작업은 `try...catch`로 감싸고, 실패 시 사용 가능한 상태 코드(HTTP 500)나 에러 메시지를 반환하도록 처리해야 합니다.

### 🔴 C. WebSocket 상태 관리 및 메모리 누수 위험
*   **문제점:** `activeDownloads` 맵을 관리하지만, 터널의 비정상 종료 시 스트림을 정리하는 로직이 엉성합니다. 또한, WebSocket의 연결이 끊어졌는지 주기적으로 확인하는 `ping/pong` 하트비트(Heartbeat) 매커니즘이 전혀 없습니다. 연결이 끊긴 줄비(좀비) 커넥션이 누적되어 메모리가 낭비될 수 있습니다.
*   **대책:** WebSocket 서버에 하트비트 주기를 설정하여 응답이 없는 클라이언트를 강제 종료(Terminate)시키는 로직을 추가해야 합니다.

## 2. `receiverScript.ts` (원격 렌더링 서버)의 주요 결함 

### 🔴 A. 쉘 명령어 치환 취약점 (Command Injection)
*   **문제점:** `exec('pkill -i -f "' + appName + '"')` 로직에서 `appName`을 그대로 쉘에 넘깁니다. 만약 누군가가 `"; rm -rf / ;"` 와 같은 요상한 페이로드를 전달한다면 수신측 Mac이 초토화될 수 있는 최악의 헛점입니다.
*   **대책:** `execFile` 이나 `spawn` 을 사용하여 쉘 명령어 인자를 배열 형태로 안전하게 전달해야 합니다. 또는 값 자체에 특수 문자가 들어오지 못하도록 정규식으로 엄격히 필터링해야 합니다.

### 🔴 B. `screencapture` 무한 루프 과부하
*   **문제점:** `setInterval`을 이용해 `screencapture`를 400ms마다 실행하는데, 명령어가 겹치거나 파일 쓰기 지연이 발생할 경우에 대한 방어벽이나 상태 잠금(Lock) 처리가 없습니다. 시스템 리소스를 갉아먹는 주범이 됩니다.
*   **대책:** 동기적으로 처리하거나 타이머 대신 재귀적인 `setTimeout`을 사용하여 앞의 캡처 작업이 완전히 끝난 후에만 다음 캡처가 예약되도록 구조를 개편해야 합니다.

### 🔴 C. 네트워크 동기화 실패 시 재시도 로직 부재
*   **문제점:** 파일 사이즈나 `mtime`이 달라서 `REQUEST_FILES`를 요청했는데, 중간에 패킷이 유실되거나 서버에서 못 보내면 무한 대기 상태에 빠집니다.
*   **대책:** 타임아웃을 걸어두고 파일을 못 받았을 경우 재요청(Retry)하는 신뢰성 있는 로직이 추가되어야 합니다.

## 3. `start.sh` / `start.bat` 의 결함
*   **문제점:** 포트(3000)를 죽이는 강력한(`kill -9`, `taskkill /F`) 접근을 사용합니다. 이는 데이터 손실을 유발할 수 있는 가장 거친 방법입니다.
*   **대책:** 부드러운 종료 신호(`SIGTERM`)를 먼저 보내고 2~3초 뒤에 살아있으면 그때 강제 종료(`SIGKILL`) 하는 안전한 포트 점유 해제 절차(Graceful Shutdown)가 필요합니다.
